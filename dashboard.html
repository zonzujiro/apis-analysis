Key Findings

1. Harmony uses 36.2% of editorx-core-api (275 out of 759 APIs).
2. FLOWS is the most consumed layer in absolute terms (75 APIs, 27.3% of Harmony's deps), reflecting heavy use of orchestration/business logic from REP.
3. EDITOR_INFRA has the highest penetration at 51.3% — Harmony uses more than half of all editor infrastructure APIs. These are foundational framework APIs (ComponentEditorAPI, StageContextBuilderAPI, TraverseDataServiceAPI, LayoutBuilderAPI, etc.).
4. SITE layer has 45.6% penetration — nearly half of all site-level APIs are consumed (DocumentServicesAPI, ExperimentsAPI, EditorPointerAPI, PreviewAPI, etc.).
5. DERIVATIVE_STATE has 39.3% penetration — Harmony relies heavily on computed state from REP (ComponentMeasureAPI, GridDerivativeStateAPI, SuperGridDerivativeStateAPI, etc.).
6. All layers except REPLUGGABLE are used — even 3 VERTICAL APIs are consumed (MultiEditor* APIs).
7. ~64% of REP APIs are unused by Harmony — 484 APIs exist only for other consumers or are unused.

Key Findings

1. EDITOR_INFRA is almost fully consumed (92.3%) — only 3 APIs excluded (ExternalComponentEditorBuilderAPI, FeaturesMapperAPI, GfppAPI). These are foundational framework APIs.
2. UI layer has the lowest coverage (41.1%) — 96 UI-layer APIs are not whitelisted. These are mostly optional indicators, panels, inspectors that Harmony replaces with its own implementations.
3. FLOWS/DATA_SERVICE/DERIVATIVE_STATE all cluster around ~76% — consistent three-quarter usage across the core business logic layers.
4. 505 whitelist APIs have no match in editorx-core-api. This number is inflated because the comparison only scanned SlotKey definitions inside editorx-core-api/src/. Many of these 505 are SlotKeys defined in other REP packages — @wix/repluggable-addons-api (PanelsAPI, HybridStoreAPI, WorkspaceAPI), @wix/repluggable-workspace, @wix/repluggable-commands, plus Private APIs and ComponentEditor behavior hooks from individual REP packages. A full scan of all REP packages would reduce the unmatched count significantly.
5. Whitelist vs Harmony source comparison: Whitelist includes 557 core-api APIs, but Harmony source only directly imports 275. The gap (~282 APIs) represents transitive dependencies — APIs pulled in automatically as deps of whitelisted entry-points.

Scope caveat: This comparison only scanned editorx-core-api/src/ for SlotKey definitions. Many of the 505 "unmatched" APIs are actually SlotKeys defined in other REP packages that editorx-core-api re-exports or that the whitelist depends on directly:

Matching Algorithm

  1. Extracted all declareAPIs and dependencyAPIs values from the lock file's metadata section (1,062 unique API names)
  2. Extracted all export const XxxAPI: SlotKey definitions from editorx-core-api .ts files, capturing both the name field and the variable name (811 total)
  3. Normalization: strip spaces, underscores, hyphens, dots, then lowercase. E.g., "Component Data API" → "componentdataapi" matches "ComponentDataAPI"
  4. Two-pass matching: first against SlotKey name field, then against variable name

Matching Quality

Verified by spot-checking ~30 suspect APIs from the 505 unmatched list. Within editorx-core-api scope, only 1 false negative: Theme API vs Site Theme API (name collision).
Known limitation: The scan only checked SlotKey definitions in editorx-core-api/src/. It did not scan repluggable-addons-api, repluggable-workspace, repluggable-commands, or other REP packages. APIs like PanelsAPI, HybridStoreAPI, WorkspaceAPI, MouseAPI are SlotKeys defined in those sibling packages. editorx-core-api sometimes re-exports them (e.g., export { HybridStoreAPI } from '@wix/repluggable-addons-api'), but re-exports don't match the export const XxxAPI: SlotKey pattern used by the scanner. A complete analysis would need to scan all REP packages or follow re-exports.

TPA Integration: Studio Pattern vs Harmony Pattern

SEO, A11Y, and CMS are currently integrated differently in Studio (REP) vs Harmony. As part of migrating core from Studio to a dedicated repo, we need to decide which pattern TPAs should follow.

Current State: Studio (REP)

TPAs are first-class packages inside the monorepo
  • editor-package-tpa — 24+ entry points spanning DATA_SERVICE, SITE, FLOWS, UI layers
  • editor-package-a11y-wizard-adapter — 9+ entry points across 4 layers, consumes 20+ core APIs directly
  • SEO logic embedded in responsive-pages package (SeoAutoRedirectPrivateAPI at FLOWS layer)
  • editor-package-tpa-events — hooks into publish, edit-mode, breakpoint changes
  • All bundled at build time with the editor, loaded synchronously

Current State: Harmony

TPAs live in separate repos, integrated via External APIs
  • TPAs in own repos: @wix/seo-repluggable, @wix/a11y-repluggable, CMS packages
  • Loaded dynamically via odeditor-package-external-packages-loaders during site-load sequence
  • Two-package pattern: odeditor-package-pages-external (public types) + odeditor-package-pages (internal implementation)
  • TPAs contribute via api.contribute*() methods, not direct API access
  • 4 existing external packages: pages, site-profile, stage-indicators, add-panel

Comparison

Aspect Keep TPAs inside core repo (Studio) External APIs (Harmony)
Development Simpler — one repo, one build, atomic changes across TPA + core, direct API access with full type safety TPA teams own their code in their own repos, independent release cadence, but harder to debug cross-repo
Bundle size All TPAs bundled = larger initial load, regardless of whether user needs them Async loading during site-load — smaller initial bundle, better TTI
API surface TPAs access entire core API surface — hard to distinguish public vs private, easy to create accidental dependencies Controlled surface via External API contracts — TPAs only access what's explicitly exposed
Ownership Core team responsible for TPA code on-call; TPA teams need core repo access TPA teams fully own their code; core team only maintains the External API contract
Coordination No cross-repo coordination needed; CI catches breaking changes immediately Version drift risk between external API and TPA implementation; need contract testing
Flexibility TPAs can use any core API method directly TPAs limited to what's designed into the External API; new TPA needs require External API changes
Boilerplate Standard entry point + API pattern Two-package pattern per integration point (types + implementation), contribution slots, bundle descriptor

Key Consideration

Surface area constraint
The External API pattern fundamentally limits what TPAs can consume. In Studio, the A11Y adapter directly uses 20+ core APIs (BreakpointsAPI, ColorPickerAPI, DialogsFlowsAPI, DocumentServicesAPI, etc.). With External APIs, it would only access what's explicitly designed into the external contract. This is both the strength (controlled, stable surface) and the weakness (anything the TPA needs must be anticipated and pre-designed into the External API).

Rules for Creating Core

Draft criteria for what belongs in the new core package, derived from architectural analysis of both REP and Harmony.

Structural Rules

1. Layer hierarchy is inviolable
APIs declare a layer: SITE → EDITOR_INFRA → DATA_SERVICE → DERIVATIVE_STATE → FLOWS → UI → VERTICAL. Dependencies only flow downward. No exceptions.
2. Every public API is a SlotKey
Interface defined separately from implementation. Published via @wix/editorx-core-api (or its successor). The SlotKey IS the contract — implementation can vary across editors.
3. Public means stable
Marking public: true is a commitment to backward compatibility. External consumers (Harmony, Santa Editor, future editors) depend on it. Breaking changes require migration path.

What Belongs in Core

4. Core = shared infrastructure, not product features
If only one vertical or TPA uses it, it's not core. Core APIs serve multiple consumers. A "core API" should have at least two independent consumers before extraction.
5. Core should be Harmony-consumed (demand-driven)
Our data shows 275 of 759 REP APIs (36.2%) are used by Harmony. The ~484 unused ones are:
Santa-only Deprecated / dead Over-extracted
For the new core, an API should have a clear consumer before extraction — no speculative APIs.
6. Method granularity matters
At 42.3% overall method usage, current interfaces are roughly 2× larger than needed. New core interfaces should be tighter:

What Does NOT Belong in Core

7. TPA-specific logic stays outside
SEO, A11Y, CMS implementations should not live in core, regardless of integration pattern. Core may provide extension points (External APIs, contribution slots), but not the TPA logic itself.
8. Component-type-specific editors stay outside
VERTICAL layer packages (button editor, box editor, section editor) are consumers, not core. They implement the ComponentEditorAPI contract but don't define it.
9. UI implementations that aren't APIs stay outside
Panel components, indicator renderers, dialog implementations — unless they expose a reusable API contract consumed by multiple packages.

Extraction Criteria

10. Demand-driven extraction
Don't extract an API into core "just in case." Extract when a second consumer needs it. The 63.8% of REP APIs unused by Harmony demonstrate the cost of speculative extraction.
11. Interface-first design
Define the SlotKey interface before writing the implementation. The interface IS the contract. Review the interface as a standalone artifact — if it doesn't make sense without reading the implementation, it needs redesign.
12. No deprecated baggage
Methods marked deprecated in REP should not migrate to the new core. Clean break. Examples from current analysis:
StageContextBuilderAPI.addPointerContextToRef StageContextBuilderAPI.getContributions EditorPointerAPI.getEditorPointerUniqueId
